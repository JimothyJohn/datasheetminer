# Cross-vendor historical-duplicate cleanup

## Goal

The `compute_product_id` family-aware fix shipped 2026-04-26 prevents
**future** prefix-drift duplicates (e.g. `MPP-1152C` vs `1152C` for
the same physical SKU). It does **not** retroactively merge rows that
were already written under different IDs. This doc plans a one-time
audit + merge for those historical orphans across all manufacturers.

## What we already know

The Parker MPP cleanup on 2026-04-26 surfaced 22 such groups inside
just one product family — three runs of the same PDF wrote the same
motor as `MPP-1152C`, `MPP1152C`, and `1152C` with different specs
extracted from different table columns. We deleted the prefix-less
rows by hand because the values were demonstrably misextractions
(peak torque mislabeled as rated torque), not legitimate divergent
data.

The same drift could hide in other families, *especially* those with:

- **Long prefixes that the LLM sometimes drops** — anything where the
  vendor's marketing prefix (`MR-J5`, `SGM7`, `BCM`, `1FK`) appears as
  a column header in the catalog and Gemini sometimes treats it as
  redundant.
- **Repeat ingests** with `--force` from before the family-aware ID
  fix — every one stamped a new UUID instead of upserting.
- **Catalogs with both compact and verbose part-number forms** on
  different pages (Mitsubishi MR-J5: `MR-J5-40G`, `J5-40G`, `40G`).

## Non-goals

- **No new ingestion logic.** The forward fix is already shipped in
  `compute_product_id`. This is purely a historical sweep.
- **No automated value-merge.** When two rows for the same SKU disagree
  on a numeric field, picking a winner is a judgement call — the Parker
  case showed peak-vs-rated confusion, not just punctuation drift.
  Anything ambiguous goes on a human-review list, not auto-resolved.
- **No prod data touch.** Run only against `dev`. Promote the cleaned
  set to staging/prod via the existing `./Quickstart admin promote`
  workflow once approved.

## Approach

Three phases, each auditable on its own.

### Phase 1 — Audit script

`cli/audit_dedupes.py` (new): scan all motor/drive/gearhead/etc rows in
DynamoDB; group by **family-aware normalized core** (the same
`_strip_family_prefix` rule the new `compute_product_id` uses); emit
JSON of every group with ≥2 rows.

For each group, the report includes:

- The normalized core (e.g. `1152c`)
- Each row's `PK`/`SK`, `part_number`, `product_family`, `manufacturer`
- A side-by-side diff of every populated field, flagging:
  - **identical** — safe auto-merge
  - **complementary** — one row has a field, the other has it null →
    safe auto-merge (take the non-null)
  - **conflicting** — both populated, different values → human review
- A suggested action: `merge`, `review`, or `delete-junk` (the
  placeholder-name pattern from the Parker case)

Output: `outputs/dedupe_audit_<timestamp>.json`.

No DB writes in this phase.

### Phase 2 — Auto-merge the safe cases

`cli/audit_dedupes.py --apply --safe-only`: for groups where every
field is `identical` or `complementary`, write the merged row under the
canonical (family-aware) UUID, then delete the orphan rows. The
canonical row uses the **most-populated** part number form as the
`part_number` field — Parker style, that's `MPP-1152C`, not `1152C`.

What "merged row" means concretely:

- Per field, take the non-null value. If both non-null and equal, use
  it. If both non-null and unequal, this group should have been
  classified `conflicting` and excluded from `--safe-only`.
- `pages` becomes the union of all source rows' page lists (so
  provenance stays).
- `datasheet_url` keeps the URL of the row with the most fields filled.

### Phase 3 — Human review queue for conflicting groups

`outputs/dedupe_review.md` (auto-generated by Phase 1 in addition to
the JSON): one section per `conflicting` group, with:

- The candidate part numbers
- A 3-column table of the disagreeing fields (`field | row A | row B`)
- A direct link to the source datasheet PDF for each row

Reviewer fills in their pick column-by-column, runs
`cli/audit_dedupes.py --apply --from-review outputs/dedupe_review.md`,
which performs the merge and writes the picks.

This is the slow part — Parker had 22 conflicting groups, and even at
30 seconds per group of manual eyeballing that's 11 minutes per
manufacturer. Across the dev DB that's probably an hour total. Not
worth automating further.

## Edge cases

- **Family is wrong.** A row's `product_family` field might itself be
  a misextraction. Audit should flag groups where the same normalized
  core appears under two different `product_family` values — those
  shouldn't merge automatically.
- **MPP vs MPJ.** The Parker case showed that the family is identical
  ("MPP" series) but the part-number variant prefix (`MPP-` vs `MPJ-`)
  marks a physically different motor. The audit must respect the
  *full* prefix, not just `product_family` — if a row carries
  `part_number=MPJ-1152C`, normalize to `mpj1152c`, not `1152c`.
  This is the same safety guard already in `_strip_family_prefix`:
  only strip when the *exact* `product_family` token is the prefix,
  never a sibling family.
- **Datasheet URL drift.** Same physical motor, two different vendor
  catalog URLs — that's normal and not a duplicate signal. Group
  purely on `(manufacturer, normalized_part)`, ignore the URL.
- **Prod safety.** Phase 2 must refuse to run unless `--stage dev`
  is explicit; Phase 3 review file must include the stage in its
  filename so a stale file can't accidentally be applied to prod.

## Estimated effort

- Audit script (Phase 1): 2 h
- Safe auto-merge (Phase 2): 1 h
- Review-applier (Phase 3): 1–2 h
- Manual review pass over dev DB: 1 h, depending on how many
  manufacturers have drift

Half a day for the code, an hour for the human review. Worth doing
**before** any large-scale promotion to prod, because cleaning up dev
once is a lot cheaper than cleaning up prod and dev separately.

## Triggers

Surface this doc when the current task touches any of:

- `cli/admin.py:purge`, `cli/admin.py:promote`, manual deletes against
  DynamoDB
- A new vendor ingestion where the catalog uses inconsistent
  prefix forms (Mitsubishi MR-J5, Yaskawa Σ-7, Siemens 1FK)
- User mentions "duplicate", "dedupe", "merge rows", "same product
  twice", "two part numbers for one motor"
- Promotion to staging/prod (`./Quickstart admin promote`) — flag this
  as a pre-flight to surface lurking duplicates before they propagate
- `datasheetminer/ids.py:compute_product_id` or
  `_strip_family_prefix` modifications that might invalidate prior
  IDs and create a fresh wave of duplicates
